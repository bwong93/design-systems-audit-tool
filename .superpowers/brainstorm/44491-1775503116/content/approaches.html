How should score context appear in the report?

Three approaches to educating, driving action, and creating accountability — pick the one that fits best

A

Inline taglines

Each score card gets a one-line "what this means" statement directly beneath the grade. No separate section — context travels with the number.

Parity Score
78
Good
Minor drift. Engineers are implementing design intent accurately for most components — remaining gaps are unlikely to block users.
✓ Contextual ✓ Compact ✗ No action guidance
B

Score legend section

A "Reading this report" section at the top explains each score once: what it measures, what causes it to drop, and what each team should do about it.

Reading this report
Parity Score
How closely code reflects the Figma spec. Low = designers and engineers are working from different sources of truth.
⚙ Engineer action when low
Token Score
% of components using design tokens vs hardcoded values. Low = rebranding will require manual fixes everywhere.
⚙ Engineer action when low
✓ Educational ✓ Clear ownership ✗ More scrolling
C

Hybrid — tagline + grade legend

Each score card has a short impact tagline. Below the scores, a compact grade legend maps Excellent → Critical to a plain-english business consequence and one action per team.

Parity Score — How closely code reflects the Figma spec
Excellent 90–100 Tightly aligned. Safe to reference Figma for implementation.
Good 75–89 Minor drift. Batch fixes in next sprint.
Fair 60–74 Noticeable gaps. Schedule a remediation sprint.
Poor 40–59 Figma is no longer a reliable handoff source.
Critical 0–39 System trust at risk. Requires immediate joint action.
✓ Educates + drives action ✓ Shared ownership ✓ Reusable per score
